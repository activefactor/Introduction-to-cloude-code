# ClaudeCode初心者向けオンボーディングサイト 要件定義書

## 1. 文書の目的

本書は、ClaudeCodeをこれから使う人向けのオンボーディングサイトを制作するための要件を定義するものです。

対象読者は、Chat AIで調べ物・要約・壁打ちなどをしているが、ClaudeCodeは未経験または触り始めの人です。大学生でも理解できる平易さを保ちつつ、ITに抵抗のない読者が「導入」「理解」「活用開始」まで進めることを目標とします。

## 2. サイトの目的

- ClaudeCodeが「実際にファイルを作る・編集する」体験価値を最初に伝える
- ClaudeCodeの基本概念をやさしく理解してもらう
- Chat AIとの違いを明確に理解してもらう
- 自分に合った導入方法を選べるようにする
- 便利な使い方を具体的にイメージしてもらう
- 料金、制限、セキュリティ面の不安を減らす

## 3. 想定ターゲット

### 主なユーザー像

- Chat AIを普段使っている
- IT知識は初級から中級
- 開発環境に強くはないが、ツール利用には抵抗がない
- コーディングの経験は問わない
- 学生、若手エンジニア、IT職、情報感度の高い社会人

### ユーザーの課題

- ClaudeCodeで何ができるのか分からない
- どの導入方法を選べばいいか分からない
- 専門用語が難しそうで不安
- 費用や安全性に不安がある
- Chat AIと比べて何が嬉しいのか腹落ちしていない

## 4. 企画方針

### 基本方針

- 体験から入って、あとから理解を深める
- 初心者の不安を減らす
- 先に全体像、その後に詳細を見せる
- 抽象概念は具体例とセットで説明する
- 難しい言葉には短い言い換えを添える
- 導入判断に必要な情報を偏りなく提示する
- 解説と具体的なステップを交互に配置し、読むリズムに緩急をつける

### トーン&マナー

- やさしい
- 断定しすぎない
- できるだけ平易
- ただし内容は薄くしない
- 読者を初心者扱いしすぎず、丁寧に伴走する

## 5. スコープ

### 本サイトで扱う内容

- ClaudeCodeの概要
- Chat AIとの違い
- 導入方法の比較
- 代表的な活用事例
- 料金体系
- モデルの使い分け
- セッションの考え方
- 短期記憶と長期記憶
- トークンと上限の考え方
- ルールの意味と書き方
- スキルの意味と便利な事例
- セキュリティ
- オプトアウト
- サンドボックス
- 関連用語集

### 本フェーズで扱わない内容

- 実際のログイン情報や個別契約手順の詳細
- リアルタイムに変動する価格や制限値の断定的記載
- 高度なAPI実装や企業向け運用設計
- ClaudeCode以外の多数ツール比較の深掘り

## 6. 情報設計要件

### 必須ページ

1. トップページ
2. まず体験してみる
3. ClaudeCodeとは
4. Chat AIとの違い
5. 導入方法の比較
6. はじめての使い方
7. 便利な活用事例
8. 料金とトークン
9. モデルの使い分け
10. セッション・記憶・ルール・スキル
11. セキュリティ・オプトアウト・サンドボックス
12. 用語集
13. FAQ

### 推奨導線

- トップページから「まず体験する」「まず知る」「導入方法を選ぶ」「使い方を見る」に分岐できること
- 各ページ末尾に次に読むべきページへの導線を置くこと
- 途中離脱しても要点が拾えるよう、各ページ冒頭に要約を置くこと
- 解説ページの直後に実践ページ、または実践セクションを配置し、学習テンポに緩急をつけること

## 7. コンテンツ要件

### 7-0. 体験先行ページ

必須要件:

- ClaudeCodeがファイルを作る、または編集する体験を最初に見せる
- 「指示しただけで実作業が進む」ことを視覚的に理解できる構成にする
- 詳しい説明より先に、驚きと興味を作る

表現要件:

- ステップ表示
- ビフォーアフター
- フロー図

### 7-1. ClaudeCodeとは

必須要件:

- ClaudeCodeを一言で説明する
- Chat AIとの違いの入口になる説明を置く
- 「会話するだけでなく、ファイルや作業に関われる」点を分かりやすく示す

### 7-2. Chat AIとの違い

必須要件:

- チャット中心のAIとClaudeCodeの違いを比較表で示す
- 特に「ファイルを読める・作れる・編集できる」点を中心に解説する
- 調べ物、要約、壁打ちとの違いを実例で示す
- 「何ができるようになるか」を利用者視点で説明する

比較軸の例:

- できること
- 実ファイルとの関わり
- 継続作業への向き不向き
- 再現性
- 開発作業との相性

### 7-3. 導入方法の比較

必須要件:

- Desktopアプリ
- CLI
- VS Code + 拡張

各方式で記載する内容:

- 向いている人
- メリット
- デメリット
- おすすめ度
- 最初に選ぶならどれか

### 7-4. 活用事例

必須要件:

- 3件以上の具体事例を掲載する
- 各事例で「従来」と「ClaudeCode活用後」を比較する
- 技術者でなくてもイメージしやすい内容を含める

事例候補:

- レポートやREADMEのたたき台作成
- 小さな自動化スクリプト作成
- 既存コードの理解
- バグの切り分け
- ファイル整理や命名改善

### 7-5. 料金とトークン

必須要件:

- 料金体系の考え方を分かりやすく説明する
- トークンを「ざっくり何か」で説明する
- 上限到達時に何が起こるかをやさしく説明する
- 上限解除までの待ち時間の考え方を説明する

注意事項:

- 数値は変更されうるため、更新日や注記を明示できる構造にする

### 7-6. モデルの使い分け

必須要件:

- モデルごとの向き不向きを説明する
- トークン消費の目安は概念的に説明する
- 「速さ重視」「精度重視」「重い作業向け」などの選び方を示す

望ましい表現:

- 迷ったらこれ、という初心者向けガイド
- 作業内容別おすすめ一覧

### 7-7. セッション・記憶・ルール・スキル

必須要件:

- セッションの意味
- 短期記憶と長期記憶の違い
- ルールを書く意味
- スキルとは何か

各項目で求めること:

- 一言での説明
- 初心者向けのたとえ
- 実際に便利になる例

### 7-8. セキュリティとサンドボックス

必須要件:

- 何が危険になりうるかを脅しすぎず説明する
- オプトアウトの考え方を初心者にも分かるように説明する
- サンドボックスの役割を説明する
- 任せてよい作業と慎重に扱う作業を分けて示す
- 機密情報や公開前情報の扱いに注意を促す

オプトアウトで含める内容:

- オプトアウトとは何か
- どのような不安に関係するのか
- 設定や方針は将来変わる可能性があるため、最新情報確認が必要であること
- 初心者向けには「迷ったらまず確認する項目」として案内すること

### 7-9. 用語集

必須要件:

- AIエージェント
- エージェントループ
- MCPサーバー
- セッション
- コンテキスト
- トークン
- サンドボックス
- オプトアウト
- ルール
- スキル

要件:

- 1用語あたり短く簡潔に説明する
- 可能なら関連ページへのリンクを置く

### 7-10. FAQ

必須要件:

- 初心者が抱きやすい質問を掲載する
- 5件以上

質問例:

- まず何から始めればよいですか
- プログラミング経験が少なくても使えますか
- Chat AIと何が違いますか
- 課金はどのくらい気にすべきですか
- 勝手に危ない操作をされませんか
- オプトアウトは気にしたほうがよいですか

## 8. UI/UX要件

### 全体要件

- PCとスマートフォンの両方で読みやすい
- 長文でも疲れにくい
- 初学者が迷子になりにくい
- 比較しやすいレイアウトを採用する
- 体験パートと解説パートの切り替わりが分かりやすい

### 表現要件

- 比較表を使う
- 図解やステップ表示を使う
- 難語には補助説明を付ける
- 注意点は視覚的に区別する
- 重要ポイントは短く要約する
- 体験を見せる図解を冒頭導線に配置する
- 長い説明が続かないよう、解説と実践の表示リズムをつくる

### デザイン要件

- 親しみやすく、信頼感がある
- 技術系すぎて冷たくならない
- 子どもっぽくはしない
- 情報の優先順位が視覚的に分かる
- 体験パートでは、行動と結果が一目で分かる視覚設計にする

## 9. 機能要件

### 必須機能

- ページ内ナビゲーション
- 主要ページ間リンク
- レスポンシブ対応
- 用語や補足情報を見つけやすい導線
- 図解やステップコンテンツをスマホでも見やすく表示できること

### あると望ましい機能

- 検索機能
- 用語ポップオーバー
- おすすめの読み順表示
- 更新日表示

## 10. 非機能要件

- 表示が軽いこと
- 静的サイトとして運用しやすいこと
- コンテンツ更新がしやすいこと
- 長文でも視認性が落ちにくいこと
- 将来ページ追加しやすい構成であること

## 11. ライティング要件

- 中学生後半から大学生でも読める文体を意識する
- 1文を長くしすぎない
- 専門用語には言い換えを添える
- 断定が難しい情報には注記を添える
- 「知らないと読めない説明」を避ける
- まず体験を見せ、そのあとに説明する順番を基本とする

## 12. 更新運用要件

更新対象になりやすい情報:

- 料金
- モデル名
- トークン上限
- 制限解除の仕様
- 導入手順
- オプトアウトに関する設定や説明

要件:

- 更新日を明示できること
- 可変情報と固定解説を分けて管理できること
- 将来的にCMSなしでも更新可能な構成にすること

## 13. 成功指標案

- 初見ユーザーがClaudeCodeの概要を説明できる
- 自分に合う導入方法を選べる
- Chat AIとの違いを1分で説明できる
- 不安点として多い料金・安全性・制限を理解できる
- 読後に最初のユースケースを1つ試せる

## 14. 制作フェーズで次に必要なもの

次工程では以下を作成する。

1. サイトマップ
2. ワイヤーフレーム
3. 各ページの見出し構成
4. 初稿テキスト
5. デザイン方針

## 15. 備考

ClaudeCodeに関する仕様のうち、料金や制限のような変動可能性がある項目は、制作時点での最新情報を確認して反映する必要があります。そのため、本文中では「考え方を理解できる説明」と「最新情報への導線」を分けて設計することが望ましいです。
