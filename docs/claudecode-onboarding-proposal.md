# ClaudeCode初心者向けオンボーディングサイト 提案書

## 1. 提案概要

ClaudeCodeをこれから使い始める人に向けて、やさしく・実用的に・不安を減らしながら導入できるオンボーディングサイトを制作することを提案します。

想定読者は、ITリテラシーはある程度あるものの、主にChat AIを「調べる・まとめる・壁打ちする」用途で使ってきた人です。大学生でも読み進められる平易さを保ちつつ、「ClaudeCodeを使うと何が変わるのか」「どこが便利で、どこに注意が必要か」を具体例ベースで理解できる構成にします。

## 2. 背景と課題

Chat AIに慣れている人でも、ClaudeCodeに触れると次のような戸惑いが起こりやすいです。

- チャットとの違いが直感的に分からない
- Desktop、CLI、VS Code拡張など、導入方法が複数あり選びにくい
- セッション、トークン、ルール、スキル、サンドボックスなど新しい用語が多い
- 「便利そう」ではなく「自分がいつ使うのか」がイメージしづらい
- セキュリティや課金に不安がある

このため、単なる機能紹介ではなく、導入判断から最初の成功体験までを支援する説明設計が必要です。

## 3. サイトの目的

本サイトの目的は次の3つです。

- ClaudeCodeの「実際にファイルを作れる・操作できる」体験価値を最初に伝え、興味を引き上げる
- ClaudeCodeの全体像を、初心者にも分かる言葉で短時間に理解してもらう
- 自分に合った導入方法と使い方を選べるようにする
- 最初の不安を減らし、「まず試してみよう」と思える状態をつくる

## 4. ターゲット像

### 主対象

- Chat AIを日常的に使っている人
- エンジニア、情報系学生、IT職種、ノーコード/ローコード利用者
- コードは少し読める、またはAIに書かせたことはある
- 開発環境やエディタの名前は知っているが、CLIには強くない

### 理解レベルの前提

- 専門用語は最初から知っている前提にしない
- ただし、説明しすぎて冗長にならないよう「まず直感」「必要なら詳細」の2段階で見せる

ご提示いただいた項目は、初心者がつまずくポイントをかなり広く押さえています。以下の形で再編すると、理解しやすくなります。

### E. 安全に使う章

## 6. 追加で提案したいコンテンツ

元案に加えて、以下のコンテンツがあるとオンボーディングサイトとしてかなり強くなります。

### 6-1. 「最初の3ユースケース」

初心者は機能説明よりも、最初に何を試せばよいかが分かると前に進みやすいです。たとえば以下です。

- 既存ファイルの要約と改善提案
- 小さなスクリプト作成
- READMEやドキュメント作成

### 6-2. 失敗しにくいプロンプト例

- 良い依頼の仕方
- 曖昧な依頼との違い
- 段階的に依頼する例

### 6-3. 「任せてよい作業 / 人が確認すべき作業」

AI活用の安心感に直結します。

- 任せやすい: 下書き、比較、整形、簡単なコード生成
- 確認必須: 削除、課金、セキュリティ、公開前の内容

### 6-4. よくあるつまずき集

- 思った通りに動かない
- 途中で文脈がずれた
- ファイルを触ってほしくない
- トークンが足りない
- どのモデルを選ぶべきか迷う

### 6-5. 用途別おすすめスタートガイド

- 学生向け
- 非エンジニア職向け
- エンジニア向け

### 6-6. オプトアウトのやさしい説明

初心者にとっては「何が学習や改善に使われるのか」「自分で選べるのか」が不安になりやすいため、独立した説明があると安心感が出ます。

- オプトアウトとは何か
- 何をオフにする話なのか
- どんな人が気にしておくべきか
- 利便性とのバランスをどう考えるか

## 7. 企画コンセプト

### コンセプト案

「まず感動する。だから理解したくなる。ClaudeCodeのやさしい入門」

### 目指す体験

- 最初に「え、そこまでできるの？」という驚きがある
- 読みながら不安が減る
- 難しい言葉も怖くない
- 自分でも使えそうだと感じる
- 読後に最初の一歩が分かる

## 8. 伝え方の方針

初心者向けで重要なのは、正確さだけでなく「分かる順番」です。次の方針を推奨します。

- 最初に「便利そう」ではなく「実際にできる」を見せる
- 最初に定義を置きすぎず、先にイメージを伝える
- 専門用語は「ひとことで言うと」で要約してから説明する
- 比較表を多用する
- 抽象説明だけで終わらせず、必ず具体例を添える
- 「便利」だけでなく「注意点」も同じ画面で示す
- 1ページあたりの情報量を詰め込みすぎない

### 8-1. 体験先行の見せ方

初学者向けでは、最初に機能一覧を見せるより、「1回の指示でファイルができる」「修正案だけでなく実際に反映できる」といった体験を見せた方が、理解の入口として強く機能します。

そのため、サイト冒頭では以下の流れを推奨します。

1. ClaudeCodeが実際にファイルを作る・編集する体験を見せる
2. Chat AIとの違いを言語化する
3. どうやって始めるかを案内する
4. 仕組みや注意点をあとから理解してもらう

### 8-2. コンテンツの緩急

説明だけが続くと理解は深まっても、興味が落ちやすくなります。逆に体験例だけが続くと、仕組みが分からず不安が残ります。

そのため、以下のように「解説」と「具体ステップ」を交互に配置する構成を提案します。

- 解説: 何がすごいのかを短く理解する
- 実践: 実際にどう使うのかをステップで見る
- 解説: その裏にある考え方を知る
- 実践: もう少し進んだ使い方を試す

このリズムにすることで、理解と興味の両方を維持しやすくなります。

## 10. 表現方法の提案

テキストだけでなく、次の表現を組み合わせると理解しやすくなります。

- 「指示 → 実行 → ファイルができる」の体験図
- 比較表
- アイコン付きのメリット・デメリット整理
- 「従来のやり方 vs ClaudeCode」の並列表現
- 1ステップずつ進む図解
- 用語のツールチップや補足カード
- よくあるミスを示す注意ボックス
- 「オプトアウトって何？」のような不安解消カード

### 10-1. 解説図の構成提案

サイト内で使う図解は、文章の補助ではなく「ひと目で腹落ちすること」を目的に設計するのが効果的です。特に以下の図を用意すると、理解が一気に進みます。

#### 図1. Chat AIとClaudeCodeの違い

- 左: Chat AIは会話中心
- 右: ClaudeCodeは会話しながらファイルや作業にも関われる

#### 図2. ClaudeCodeで起きること

- 人が依頼する
- ClaudeCodeが内容を読む
- 必要に応じてファイルを作る・編集する
- 人が確認して次に進む

#### 図3. 解説と実践が交互に進む学習導線

- 理解する
- 小さく試す
- なぜそうなるか知る
- 次の一歩を試す

#### 図4. セッションと記憶のイメージ

- 今の会話で覚えていること
- 続けて使うと効いてくるルールや設定

#### 図5. 安全に使うための考え方

- 任せやすい領域
- 確認しながら進める領域
- 慎重に扱う領域

## 14. 補足

料金体系、モデル名称、トークン上限などは変更される可能性があるため、制作時は「更新前提の情報」として管理する設計が望ましいです。固定文で埋め込むより、更新しやすい構造にしておくことを推奨します。
